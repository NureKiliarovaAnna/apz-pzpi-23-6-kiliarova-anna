3 ОПИС ВИКОНАНОЇ РОБОТИ
	3.1 Загальні відомості про систему
     SoundCloud — це цифрова платформа для публікації, зберігання, каталогізації, пошуку та стримінгу аудіоконтенту . Система підтримує web- і mobile-клієнти, а також надає API для партнерських інтеграцій.
     Основні функції платформи включають завантаження аудіотреків, обробку медіафайлів, управління метаданими, пошук контенту, рекомендаційні механізми та аналітику для авторів.
     Користувачами системи є артисти, слухачі, партнери та внутрішні команди, які забезпечують розвиток і підтримку платформи .
     3.2 Архітектурний стиль системи
     Архітектура SoundCloud є гібридною та поєднує мікросервісний підхід із багаторівневою організацією компонентів .
     Система еволюціонувала від монолітного Ruby on Rails застосунку до сервісно-орієнтованої архітектури шляхом поступового винесення функціональності у незалежні сервіси.
     Основними архітектурними підходами є:
- використання BFF (Backends for Frontends) для різних типів клієнтів,
- розділення логіки на рівні агрегації, доменів та базових сервісів,
- застосування подієво-орієнтованої взаємодії через Kafka,
- використання асинхронних pipeline для обробки даних.
     Такий підхід дозволяє досягти високої масштабованості, незалежності сервісів і гнучкості розвитку системи.
     3.3 Основні компоненти системи
     Архітектура SoundCloud складається з декількох логічних рівнів :
- Клієнтський рівень включає web- і mobile-додатки, а також API для партнерів;
- Edge/BFF-рівень забезпечує взаємодію клієнтів із системою, виконує автентифікацію, обмеження запитів і кешування;
- Value-Added Services реалізують агрегацію даних і застосування бізнес-логіки;
- Domain Gateways централізують доступ до доменних операцій і зменшують кількість міжсервісних викликів;
- Foundation Services надають базові API для роботи з сутностями, такими як користувачі, треки та плейлисти;
- Data layer включає MySQL як основне сховище, Kafka як подієвий брокер, Elasticsearch для пошуку та кешуючі системи;
- Інфраструктурний рівень представлений Kubernetes, HAProxy та Zipkin для забезпечення стабільності й спостережуваності системи.
     3.4 Взаємодія компонентів
     Взаємодія компонентів системи реалізується через поєднання синхронних і асинхронних механізмів .
     При оновленні треку клієнт надсилає HTTP-запит через BFF-рівень. Запит передається через Domain Gateway до відповідного foundation service, де зміни зберігаються в MySQL.
     Після цього подія про зміну публікується в Kafka через Event Feeder. Далі Indexer обробляє подію, формує документ і передає його до Elasticsearch через Shipper для оновлення пошукового індексу.
     Таким чином, write-операції виконуються синхронно, а read-модель оновлюється асинхронно, що дозволяє розділити навантаження та підвищити продуктивність системи.
     3.5 Масштабованість та надійність
     Масштабованість SoundCloud досягається за рахунок мікросервісної архітектури та використання контейнеризації через Kubernetes.
     Система підтримує сотні мікросервісів і десятки BFF-компонентів, що дозволяє масштабувати окремі частини незалежно.
     Надійність забезпечується за рахунок використання таких технологій і підходів:
- Kafka для асинхронної взаємодії між сервісами;
- circuit breaker pattern для ізоляції збоїв;
- retry logic і timeout management;
- distributed tracing через Zipkin;
- балансування навантаження через HAProxy.
     Розділення write- та read-моделей через Kafka і Elasticsearch дозволяє зменшити зв’язність між компонентами та підвищити стійкість системи .
     3.6 Приклади програмної реалізації
     У системі SoundCloud широко використовується агрегація даних на рівні BFF. Наприклад, запит до BFF може об’єднувати інформацію про трек, політику доступу та автора з різних сервісів у єдиний DTO для клієнта.
     Також застосовується подієво-орієнтований підхід для побудови пошукового індексу: при зміні даних формується подія, яка передається через Kafka та обробляється сервісами індексації, після чого результати зберігаються в Elasticsearch.
     Це дозволяє будувати read-модель незалежно від основного сховища та забезпечує ефективний пошук і масштабування.


ВИСНОВКИ
	У результаті виконання роботи було досліджено архітектуру програмної системи SoundCloud. Встановлено, що система використовує гібридну мікросервісну архітектуру з чітким розділенням рівнів відповідальності. Проаналізовано основні компоненти системи, механізми взаємодії між ними та підходи до побудови асинхронних pipeline для обробки даних. Оцінено переваги архітектурних рішень, серед яких ключовими є масштабованість, незалежність сервісів, зниження зв’язності та підвищення надійності системи. Отримано практичні навички аналізу архітектури сучасних розподілених систем.
